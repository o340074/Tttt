# 09 — Безопасность и правовое

## Безопасность приложения

### Аутентификация и сессии
- Пароли — **argon2id** (или bcrypt cost ≥ 12).
- JWT: короткоживущий access (15 мин) + refresh (в HTTP-only, Secure, SameSite cookie).
- Ротация refresh-токенов, чёрный список в Redis при logout/компрометации.
- 2FA (TOTP) для аккаунтов — P1, обязательна для admin/support.
- Защита от brute-force: rate limiting + прогрессивные задержки на `/auth/login`.

### Авторизация
- **RBAC**: роли user / support / admin; гварды на всех `/admin/*`.
- Проверка владения ресурсом (заказ, тикет, выдача) на каждом доступе.
- Принцип наименьших привилегий для операторов.

### Защита данных
- **Шифрование секретных payload** (стоки, выданные данные) — AES-256-GCM,
  ключ в KMS / переменных окружения, ротация ключей.
- Шифрование в покое для БД (на уровне провайдера/диска) + TLS в транзите.
- Маскирование секретов в логах; секреты никогда не в логи/Sentry.
- Доступ к расшифровке — только владелец + аудит-запись.

### Безопасность API
- Валидация и санитизация всех входов (Zod/class-validator).
- Rate limiting по IP/пользователю на чувствительных эндпоинтах.
- CORS — только доверенные origin фронта.
- Защита от CSRF для cookie-сессий (SameSite + CSRF-токен где нужно).
- Заголовки безопасности (CSP, HSTS, X-Frame-Options) через Nginx/Helmet.
- Идемпотентность и проверка подписи вебхуков оплаты.

### Финансовая целостность
- Двойная запись (ledger), сверка баланса, транзакции БД для всех движений.
- Идемпотентные зачисления/выдачи (защита от двойных операций).
- Аудит-лог всех админ-действий (корректировки баланса, выдачи, возвраты).

### Инфраструктура
- Секреты — в менеджере секретов, не в репозитории.
- Регулярные бэкапы БД + проверка восстановления.
- Мониторинг (Sentry, метрики), алерты на аномалии оплат/выдач.
- Обновление зависимостей, SCA (проверка уязвимостей пакетов) в CI.
- Разделение окружений dev/stage/prod.

## Антифрод
- Лимиты на частоту пополнений/заказов, флаги подозрительной активности.
- Отслеживание мультиаккаунтинга (fingerprint, IP-паттерны) — базово в MVP.
- Ручная модерация спорных заказов через админку.

## Правовые и комплаенс-аспекты

> ⚠️ Это не юридическая консультация. Перед запуском необходима консультация с
> юристом по юрисдикции ведения бизнеса.

### Бренд и оформление
- Вдохновляемся визуальным языком (Material Design), но **не используем** логотипы,
  бренд-ассеты Google и **не выдаём себя за Google**. Название — собственное
  (напр. **AdsMarket**). Иначе — риск нарушения товарных знаков.

### Тематика товара
- Продажа рекламных аккаунтов может противоречить правилам платформ (Google Ads
  ToS) и вызывать вопросы у платёжных провайдеров и хостеров. Последствия:
  - блокировки мерчант-аккаунтов и платёжных шлюзов;
  - требования KYC/AML;
  - региональные ограничения.
- **Рекомендации:**
  - выбрать платёжного провайдера и хостинг, допускающих тематику;
  - подготовить юридические документы (Terms of Service, Refund Policy, Privacy);
  - явно описать, что именно продаётся и на каких условиях;
  - предусмотреть возрастные/региональные ограничения при необходимости.

### Данные пользователей (Privacy)
- Минимизация собираемых персональных данных.
- Политика конфиденциальности, согласия, право на удаление (по применимым нормам,
  напр. GDPR при работе с EU).
- Безопасное хранение и удаление данных по истечении необходимости.

### Финансовый комплаенс
- Учёт крипто-операций, при необходимости — KYC/AML для крупных сумм.
- Прозрачные условия возвратов и споров.

### Прогрев, антидетект-браузеры (Octo) и прокси — отдельные риски

Модель «аккаунты под заказ с прогревом» и выдача с Octo-профилем + прокси повышает
правовые и платёжные риски:
- Использование антидетект-браузеров и прокси для аккаунтов может трактоваться как
  обход правил платформ (Google) и повышать внимание провайдеров/регуляторов.
- Google-аккаунты разработчика для расширений — зона повышенного риска (потенциальное
  злоупотребление при публикации расширений).

**Позиция по реализации (важно):** платформа реализует только **бизнес-логистику** —
каталог, заказы, оплату, очереди задач, инвентарь прокси/профилей, трекинг этапов,
сборку и выдачу комплекта, аудит. Она **не** содержит и не должна содержать средств
автоматического обхода антифрод-систем, автоматизации действий внутри аккаунтов или
техник уклонения от детектирования. Прогрев и провижининг выполняют операторы —
платформа это лишь учитывает.

**Рекомендации:**
- Обязательная юр-консультация по юрисдикции до запуска; подготовить ToS/Refund/Privacy,
  явно описывающие предмет и условия.
- Шифровать все секреты (аккаунты, прокси-доступы, ссылки на экспорт профилей),
  доступ — по праву и с аудитом.
- Хранить связки ресурс↔заказ для поддержки/замены, но с минимизацией и сроками хранения.
- Проверить, что платёжный провайдер и хостинг допускают тематику.

## Чек-лист перед запуском (безопасность)
- [ ] Все секреты вынесены из кода.
- [ ] Шифрование payload включено и протестировано.
- [ ] Rate limiting и RBAC покрывают чувствительные пути.
- [ ] Вебхуки проверяют подпись и идемпотентны.
- [ ] Бэкапы и восстановление проверены.
- [ ] Юридические документы (ToS/Privacy/Refund) готовы.
- [ ] Платёжный провайдер и хостинг подтвердили допустимость тематики.
- [ ] Проведён базовый security-review (см. `/security-review`).
