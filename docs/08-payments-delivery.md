# 08 — Оплата и доставка

## Способы оплаты (MVP)

1. **Внутренний баланс (кошелёк)** — основной способ оплаты заказов.
2. **Пополнение баланса криптовалютой** — через крипто-эквайринг.

(Прямая оплата заказа криптой без пополнения — P1. Карты — P2, отдельная оценка.)

### Почему баланс + крипта
- Крипта — без чарджбэков, международная, подходит тематике.
- Баланс — удобно для повторных покупок, снижает число платёжных операций и
  комиссий, ускоряет оформление (списание мгновенное).

## Крипто-эквайринг

**Кандидаты провайдеров:** Cryptomus, NOWPayments, Coinbase Commerce.
Критерии выбора (решается в начале сессии разработки платежей):
- поддержка USDT (TRC-20/ERC-20), BTC, ETH;
- вебхуки с подписью;
- приемлемые комиссии и лимиты;
- отсутствие KYC для мелких сумм / приемлемые требования;
- стабильность и репутация.

### Поток пополнения (детально)
```
1. POST /wallet/topups { amount, asset }.
2. Создаём TopUp(status=pending, externalId=null пока не получен от провайдера).
3. Вызываем API провайдера → получаем payment address / URL + externalId.
4. Показываем пользователю QR/адрес и таймер.
5. Пользователь платит.
6. Провайдер шлёт вебхук POST /webhooks/payments/:provider.
7. Воркер:
   a. Проверяет подпись вебхука.
   b. Находит TopUp по externalId; если уже paid → выходит (идемпотентность).
   c. В транзакции БД: TopUp=paid + LedgerEntry(credit) + User.balance += amount.
8. Уведомление пользователю; UI обновляет статус (поллинг/websocket).
```

### Идемпотентность и надёжность
- Уникальный `externalId` на пополнение → повторный вебхук не задваивает зачисление.
- Все зачисления — в транзакции БД вместе с ledger-записью.
- Вебхук отвечает 200 только после успешной записи; иначе провайдер повторит.
- Просроченные pending-пополнения закрываются по TTL воркером.

## Оплата заказа с баланса
```
1. POST /orders/checkout.
2. Проверка наличия и резерв стока (для auto-позиций).
3. Проверка баланса ≥ total.
4. В транзакции БД: LedgerEntry(debit) + User.balance -= total + Order=paid.
5. Запуск выдачи (см. ниже).
   При недостатке баланса → ошибка INSUFFICIENT_BALANCE, резерв снимается.
```

## Доставка (гибридная)

Тип выдачи задаётся на уровне варианта (`ProductVariant.deliveryType`).

### A. Авто-выдача из стока (`auto`)
- У варианта есть пул `StockItem` со статусом `available`.
- **Резерв:** при оформлении единицы помечаются `reserved` (+ TTL в Redis), чтобы
  не продать одну единицу дважды.
- После оплаты: `reserved → sold`, создаётся `Delivery`, payload расшифровывается
  и показывается покупателю в ЛК.
- Если доступного стока меньше, чем нужно → недостающие позиции переводятся в
  ручную выдачу (заявка) либо заказ отклоняется (настройка политики варианта).

### B. Ручная выдача под заказ (`manual`)
- После оплаты создаётся задача выдачи (`OrderItem.deliveryStatus = awaiting_manual`).
- Задача попадает в очередь админки/поддержки.
- Оператор вносит данные → `Delivery(type=manual)`, статус `delivered`.
- Покупатель получает уведомление и доступ к данным в ЛК.
- **Политика средств:** средства списываются сразу; при невозможности выдать в
  срок — автоматический/ручной возврат на баланс (refund → ledger credit).

### Гарантийная замена (P1)
- В течение гарантийного окна (напр. 24–72 ч) покупатель может запросить замену
  бракованной позиции.
- Запрос → проверка → выдача новой единицы (`Delivery(type=replacement)`) либо
  возврат на баланс.

## Хранение и выдача секретных данных
- `StockItem.payload` и `Delivery.payload` — **шифруются** на уровне приложения
  (см. [09](./09-security-compliance.md)).
- Расшифровка — только на сервере, только для владельца заказа, доступ логируется.
- В UI — маскирование до явного «показать», кнопки «копировать»/«скачать».

## Возвраты и споры
- Возврат — только на внутренний баланс (не на крипто-адрес), через ledger.
- Все возвраты фиксируются с причиной и оператором (аудит).
- Споры ведутся через тикет-систему, привязанную к заказу.

## Комиссии и учёт
- Валюта учёта — единая (напр. USD); крипта конвертируется по курсу на момент оплаты.
- Комиссия эквайринга фиксируется в записи пополнения для сходимости.
