## Компоненты

### 1. Web (React SPA)
- Рендерит витрину, ЛК, админку.
- Общается с API только по REST (JSON), токены в HTTP-only cookie / памяти.
- i18n на клиенте (react-i18next), язык из настроек профиля / браузера.

### 2. API (NestJS)
Модульная структура. Основные модули:
- **auth** — регистрация, вход, JWT, 2FA, восстановление.
- **users** — профиль, роли, баланс (чтение).
- **catalog** — категории, товары, варианты (SKU), поиск, фильтры.
- **stock** — единицы стока, импорт, резерв, статусы.
- **orders** — корзина, оформление, статусы заказов.
- **payments** — пополнение, крипто-эквайринг, вебхуки, транзакции.
- **delivery** — авто- и ручная выдача, гарантийные замены.
- **wallet** — баланс, движения средств (ledger).
- **support** — тикеты, сообщения, вложения.
- **admin** — управление всем вышеперечисленным + статистика, аудит.
- **notifications** — email/in-app уведомления.
- **i18n** — управление переводами (при необходимости серверных).

### 3. PostgreSQL
Источник истины: пользователи, товары, стоки, заказы, транзакции, тикеты.
Финансовые операции — только в транзакциях БД (см. ниже).

### 4. Redis
- Кэш каталога и справочников.
- Хранение refresh-токенов / чёрный список.
- **Резерв единиц стока** на время оформления заказа (TTL-блокировка).
- Rate limiting (логин, оплаты, API).

### 5. BullMQ (воркеры)
Асинхронная обработка вне HTTP-запроса:
- отправка email;
- обработка вебхуков оплаты (идемпотентно);
- финализация авто-выдачи;
- напоминания, отложенные задачи (например, снятие резерва).

## Критичные потоки

### Пополнение баланса криптой
```
1. Пользователь запрашивает пополнение на сумму X.
2. API создаёт запись top-up (status=pending) и через эквайринг — платёжный адрес/ссылку.
3. Пользователь оплачивает; эквайринг шлёт вебхук.
4. Воркер проверяет подпись вебхука (идемпотентно по payment_id).
5. При подтверждении: в ОДНОЙ транзакции БД — top-up=paid + запись в ledger + рост баланса.
6. Уведомление пользователю.
```

### Оформление и авто-выдача
```
1. Пользователь оформляет заказ из корзины.
2. API проверяет наличие и РЕЗЕРВИРУЕТ единицы стока (Redis TTL + отметка в БД).
3. Списывает средства с баланса (транзакция БД: ledger + order=paid).
4. Привязывает зарезервированные единицы стока к заказу, помечает sold.
5. Отдаёт данные покупателю (в ЛК). Уведомление.
   При нехватке стока для SKU → позиция уходит в ручную выдачу (заявка).
```

### Ручная выдача под заказ
```
1. Заказ на SKU типа manual → создаётся задача выдачи (status=awaiting).
2. Средства списаны/захолдированы согласно политике (см. 08).
3. Админ/поддержка в админке выполняет выдачу, вносит данные.
4. Статус → delivered, покупатель получает данные + уведомление.
```

## Принципы

- **Идемпотентность** вебхуков и выдач (защита от двойного зачисления/выдачи).
- **Двойная запись (ledger)** для всех движений средств — аудит и сходимость.
- **Резерв стока** предотвращает продажу одной единицы двум покупателям.
- **Разделение чтения/записи** финансов через транзакции БД.
- **Stateless API** — горизонтальное масштабирование, состояние в Postgres/Redis.
