# План разработки онлайн-магазина

Статус: **эта сессия — только планирование**. Кода нет, репозиторий пуст. Документ фиксирует архитектуру,
стек и разбивку работ на фазы/сессии так, чтобы каждую последующую сессию можно было начинать с чёткого
скоупа, не пересматривая решения заново.

## 1. Цели проекта

- Полноценная, мощная витрина интернет-магазина с профессиональным дизайном (уровень крупных
  рекламных лендингов / Google Ads посадочных страниц: чистая типографика, продуманная сетка,
  анимации, высокая скорость).
- Собственный стек, без привязки к внешним no-code конструкторам.
- Деплой на собственном сервере (не Vercel/serverless-only) — значит нужен процесс, который работает
  как long-running Node-сервер + база + прокси, разворачиваемый через Docker.
- Разработка растянута на много сессий — план должен быть модульным: каждая фаза даёт рабочий
  прирост функциональности и может разрабатываться/тестироваться независимо.

## 2. Архитектура (self-hosted)

```
                        ┌─────────────────────┐
                        │        Nginx         │  TLS (Let's Encrypt), reverse proxy,
                        │  (reverse proxy)      │  статика, кэш заголовков, gzip/brotli
                        └──────────┬───────────┘
                                   │
                        ┌──────────▼───────────┐
                        │   Next.js app (Node)  │  SSR/ISR, API routes, серверные экшены
                        │   (Docker контейнер)  │
                        └───┬───────────┬───────┘
                            │           │
                 ┌──────────▼──┐   ┌────▼─────────┐
                 │ PostgreSQL  │   │    Redis      │  сессии, кэш, очереди (BullMQ),
                 │ (Docker)    │   │  (Docker)     │  ограничение частоты запросов
                 └─────────────┘   └───────────────┘
                            │
                 ┌──────────▼──────────┐
                 │  S3-совместимое      │  изображения товаров, файлы
                 │  хранилище (MinIO)   │
                 └──────────────────────┘
```

Всё поднимается через `docker-compose.yml` на своём сервере: приложение, БД, Redis, MinIO, Nginx.
CI/CD (GitHub Actions) собирает образ, гоняет тесты/линт, пушит на сервер по SSH или в приватный
registry, дальше `docker compose pull && up -d`.

## 3. Технологический стек

| Слой | Технология | Почему |
|---|---|---|
| Frontend/SSR | Next.js 14 (App Router) + TypeScript | SSR/ISR для SEO и скорости, серверные экшены, единая кодовая база с бэкендом |
| UI | Tailwind CSS + shadcn/ui (кастомизируется под собственный дизайн) | Быстрая, но не "шаблонная" вёрстка; полный контроль над стилями |
| Анимации | Framer Motion | Плавные переходы, микро-интеракции для премиального ощущения |
| БД | PostgreSQL 16 | Реляционная модель заказов/товаров/пользователей, транзакции для оплаты |
| ORM | Prisma | Миграции, типобезопасные запросы |
| Кэш/очереди/сессии | Redis + BullMQ | Кэш каталога, очереди на отправку писем/уведомлений, rate-limiting |
| Аутентификация | Auth.js (NextAuth) + свои таблицы пользователей | Email/пароль + OAuth (Google и т.п.), JWT/сессии в Redis |
| Оплата | Провайдер уточняется (ЮKassa для РФ и/или Stripe для международных платежей) | Модуль оплаты изолирован интерфейсом `PaymentProvider`, чтобы подключать несколько провайдеров |
| Файлы/изображения | MinIO (S3-совместимое) + sharp для генерации превью | Свой сервер — свой сторедж, не завязан на облако |
| Поиск | Postgres full-text search на старте → Meilisearch/Typesense при росте каталога | Не тащим Elasticsearch преждевременно |
| Мониторинг | Prometheus + Grafana (или проще: Uptime Kuma) + Sentry для ошибок | Наблюдаемость на своём сервере |
| CI/CD | GitHub Actions → SSH deploy / private Docker registry | Автоматизация выката без ручных шагов |

## 4. Модель данных (ядро)

- **User**: id, email, passwordHash, name, role (customer/admin), createdAt
- **Address**: userId, страна/город/улица/индекс, признак "по умолчанию"
- **Category**: id, slug, name, parentId (дерево категорий)
- **Product**: id, slug, title, description, price, currency, categoryId, attributes(JSON), images[]
- **ProductVariant**: productId, SKU, атрибуты (размер/цвет), цена/остаток
- **Inventory**: variantId, quantity, warehouse (на будущее — мультисклад)
- **Cart / CartItem**: привязка к сессии или к User, variantId, qty
- **Order / OrderItem**: userId, статус, сумма, адрес доставки, снапшот цены на момент заказа
- **Payment**: orderId, provider, status, externalId, сумма, вебхук-события
- **Review**: productId, userId, рейтинг, текст, модерация
- **Coupon/Discount**: код, тип скидки, условия, срок действия
- **AuditLog** (админка): кто и что изменил — важно для "мощной" админ-панели

## 5. Разбивка на фазы (сессии разработки)

Каждая фаза = одна или несколько будущих сессий с чётким deliverable.

**Фаза 0 — Фундамент**
Инициализация Next.js + TypeScript + Tailwind, Docker Compose (app/db/redis/minio/nginx),
настройка Prisma и первых миграций, CI (lint+test+build), базовый дизайн-система (цвета, типографика,
сетка, компоненты кнопок/карточек) в стиле премиальных лендингов.

**Фаза 1 — Каталог**
Категории и товары (CRUD в БД), страница списка товаров с фильтрами/сортировкой/пагинацией,
страница карточки товара, SSR/ISR для SEO, генерация превью изображений.

**Фаза 2 — Корзина и заказ**
Корзина (гость + авторизованный), оформление заказа, расчёт доставки/скидок, страница подтверждения,
письмо-уведомление о заказе (очередь через BullMQ).

**Фаза 3 — Аутентификация и личный кабинет**
Регистрация/логин (email+пароль, опционально OAuth), личный кабинет: история заказов, адреса,
избранное.

**Фаза 4 — Приём онлайн-оплаты**
Интеграция платёжного провайдера через абстракцию `PaymentProvider`, вебхуки статусов оплаты,
идемпотентность и защита от повторной обработки, страницы успеха/ошибки оплаты.

**Фаза 5 — Админ-панель**
Отдельный защищённый раздел `/admin`: управление товарами/категориями/остатками, обработка заказов
(смена статуса, возвраты), управление промокодами, базовая аналитика продаж, лог действий.

**Фаза 6 — Дизайн и UX высокого уровня**
Полировка интерфейса под "премиальный" стиль: анимации Framer Motion, адаптивность,
доступность (a11y), тёмная тема, оптимизация Core Web Vitals.

**Фаза 7 — Продвинутые фичи ("много фишек")**
Список кандидатов, приоритизируем по ходу разработки:
- Поиск с автодополнением и опечатками (Meilisearch)
- Рекомендации ("похожие товары", "с этим покупают")
- Отзывы и рейтинги с модерацией
- Wishlist / список желаний
- Промокоды и динамические скидки
- Мультивалютность / мультиязычность (i18n)
- Уведомления (email + push/telegram-бот)
- Программа лояльности / бонусные баллы
- Сравнение товаров
- SEO: sitemap.xml, structured data (JSON-LD Product/Offer), Open Graph
- A/B-тестирование лендингов (в духе "Google Ads style")

**Фаза 8 — Производительность и масштабирование**
Кэширование каталога в Redis, ISR/on-demand revalidation, оптимизация запросов Prisma, CDN для
статики/изображений, нагрузочное тестирование (k6).

**Фаза 9 — Безопасность**
Аудит по OWASP Top 10, rate-limiting, CSRF/XSS защита, ролевой доступ (RBAC) в админке, шифрование
чувствительных данных, регулярные бэкапы БД, secrets management.

**Фаза 10 — Деплой и эксплуатация**
Финальный прод-деплой: Nginx + Let's Encrypt, мониторинг (Grafana/Sentry), алерты, автоматические
бэкапы Postgres, документация по восстановлению.

## 6. Как использовать этот план в следующих сессиях

В начале новой сессии — сослаться на конкретную фазу из раздела 5 ("делаем Фазу 1 — каталог") вместо
пересказа всего проекта заново. Список "продвинутых фич" (Фаза 7) — открытый бэклог: пункты будут
добавляться/уточняться по ходу разработки.
