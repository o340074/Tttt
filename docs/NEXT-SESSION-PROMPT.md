# Стартовые промты сессий

Здесь лежит **мастер-промт (самоориентирующийся)**, **готовый промт для ближайшей
сессии** и **универсальный шаблон**. Скопируй нужный в начало новой сессии.

---

## ⭐ МАСТЕР-ПРОМТ (самоориентирующийся, рекомендуется)

> Работает на любой ветке: сам находит проект и определяет текущий эпик. Копируй целиком.

```
Ты продолжаешь разработку проекта AdVault — премиальный маркетплейс цифровых
товаров (аккаунты Google Ads: готовые из стока и под заказ с прогревом операторами).

ШАГ 0 — НАЙДИ ПРОЕКТ И СОРИЕНТИРУЙСЯ (обязательно, до любых действий):
1. Проверь, что в рабочей папке есть проект — файлы CLAUDE.md и docs/16-development-plan.md.
2. Если их нет (папка пустая / не та ветка), выполни:
     git fetch --all
     git checkout claude/advault-e2-catalog-m0omaf   # актуальная база с кодом E0+E1+E2
   (в main пока только документация/прототипы — код эпиков туда ещё не влит;
   проверяй: рядом с docs/ должны быть apps/ и packages/).
3. Прочитай для контекста, строго в этом порядке:
     - CLAUDE.md            — что строим, где что лежит, конвенции, границы;
     - docs/SESSION-LOG.md  — что уже сделано и какой эпик следующий;
     - docs/16-development-plan.md — детальный план (эпики E0–E11, вехи M0–M5);
     - раздел плана по текущему эпику + релевантные доки
       (docs/03-tech-stack, docs/04-architecture, docs/design/*, docs/backend/*).
4. Определи текущий эпик из docs/SESSION-LOG.md (блок «Текущий статус»).
   Если ещё ничего не начато — это эпик E0 «Каркас монорепо».

ЦЕЛЬ СЕССИИ — довести до готовности текущий эпик (сейчас это E3 «Кошелёк и пополнение криптой»):
  - Контракты вперёд: зафиксируй в docs/backend/{prisma-schema,openapi}.md модели
    LedgerEntry, TopUp (+IdempotencyKey) и эндпоинты кошелька/пополнения/вебхука;
    затем миграция и код;
  - apps/api: выбор крипто-эквайринга (sandbox; провайдер — за заглушкой-интерфейсом),
    POST /wallet/topups (создание пополнения: сумма, актив, адрес/QR/paymentUrl,
    Idempotency-Key), GET /wallet (баланс), GET /wallet/transactions (история),
    вебхук эквайринга с проверкой подписи; зачисление: ledger (двойная запись,
    balanceAfter) + обновление User.balance в одной транзакции БД; идемпотентность
    (TopUp.externalId @unique, @@unique(refType,refId,direction) в ledger);
    воркер/обработка просроченных pending-пополнений;
  - apps/web: экран кошелька по prototype/index.html (Wallet): баланс, форма
    пополнения (сумма, актив/сеть), экран оплаты (QR/адрес/таймер/статус,
    поллинг статуса), история транзакций; i18n EN/RU, loading/empty/error;
  - тесты: unit на ledger-сервис (зачисление, идемпотентность, сходимость баланса),
    smoke создание пополнения → вебхук → баланс вырос (фейки как в E1/E2).
  Критерий приёмки: тестовое пополнение доходит до баланса корректно и идемпотентно
  (повторный вебхук не задваивает); lint/typecheck/тесты/CI зелёные.

КОНВЕНЦИИ (из CLAUDE.md, обязательно):
  - Контракты вперёд: меняешь поведение — сначала обнови OpenAPI/Prisma в docs/backend.
  - Вертикальный срез: фронт→API→БД→тесты.
  - Дизайн только по docs/design/* и прототипу; иконки — SVG-спрайт, БЕЗ эмодзи; i18n EN/RU через ключи.
  - Деньги: decimal + транзакции + ledger + идемпотентность. Секреты: шифрование + аудит.
    RBAC на админ/операторских маршрутах; danger-confirm на необратимом.
  - Граница платформы: НЕ автоматизируем прогрев/действия в аккаунтах/обход детекта —
    только бизнес-логистика (каталог, заказы, инвентарь, очереди, трекинг, выдача, аудит).

GIT: создай feature-ветку под эпик от актуальной базы (напр. feat/E1-auth);
понятные коммиты; в конце — git push -u origin <ветка>.

ВОПРОСЫ: при любой значимой развилке (выбор библиотек, структура, спорное решение или
неоднозначность требований) — спрашивай в режиме asking (AskUserQuestion), не угадывай.

В КОНЦЕ СЕССИИ (ритуал):
  1. Обнови docs/SESSION-LOG.md: что сделано, статус эпика (E3=готово), следующий эпик (E4).
  2. Обнови статусы в docs/16 и контракты в docs/backend при изменениях.
  3. commit + push.
  4. Подготовь готовый промт следующего эпика в docs/NEXT-SESSION-PROMPT.md.
```

Для следующих эпиков просто замени в блоке «ЦЕЛЬ СЕССИИ» задачи и критерий на нужный
эпик из `docs/16` (мастер-промт сам подхватит текущий эпик из SESSION-LOG).

## ✅ ГОТОВЫЙ ПРОМТ — Сессия E3 (Кошелёк и пополнение криптой)

> Скопируй блок ниже целиком в новую сессию.

```
Продолжаем разработку проекта AdVault (маркетплейс цифровых товаров: аккаунты Google
Ads — готовые из стока и под заказ с прогревом). Готовы эпики E0 (каркас: pnpm-монорепо
apps/{web,api} + packages/{types,config}, /health, docker-compose, CI), E1 (auth:
JWT access+refresh с ротацией в Redis, argon2id, verify/reset, guard-роуты, ЛК, /me)
и E2 (каталог: Prisma-модели Category/Product/ProductVariant+Translation с
fulfillmentType/goal/tier/bundleSpec/etaMinutes/warrantyHours, публичные
GET /categories | /products | /products/:slug с фильтрами/поиском/локализацией EN-RU,
витрина/каталог/карточка на фронте, идемпотентный сидер prisma/seed.ts).

ВАЖНО — БАЗА КОДА: актуальный код лежит в ветке claude/advault-e2-catalog-m0omaf
(E0+E1+E2 ещё не влиты в main). Если в рабочей папке нет папок apps/ и packages/ — выполни:
  git fetch --all
  git checkout claude/advault-e2-catalog-m0omaf
Feature-ветку для E3 создавай от неё.

ПЕРЕД РАБОТОЙ прочитай для ориентации:
1. CLAUDE.md (корень) — контекст, конвенции, границы.
2. docs/SESSION-LOG.md — что уже сделано и что дальше.
3. docs/16-development-plan.md — раздел «E3» (и общий план/вехи).
4. docs/08-payments-delivery.md — потоки оплаты/пополнения; docs/05-data-model.md —
   LedgerEntry/TopUp; docs/backend/{prisma-schema,openapi}.md — контракты (модели и
   разделы Wallet/Webhooks уже есть в черновике — проверь и при необходимости уточни);
   prototype/index.html — эталон экрана Wallet (баланс, форма, QR/адрес/таймер).

ЦЕЛЬ ЭТОЙ СЕССИИ — реализовать эпик E3 «Кошелёк и пополнение криптой»:
- Контракты вперёд: финализируй LedgerEntry, TopUp, IdempotencyKey и эндпоинты
  кошелька в docs/backend/{prisma-schema,openapi}.md; затем schema.prisma + миграция.
- apps/api: модуль wallet — GET /wallet (баланс из User.balance, сверяемый с ledger),
  GET /wallet/transactions (история, пагинация), POST /wallet/topups (создание
  пополнения: amount, asset; Idempotency-Key; ответ с paymentUrl/address/expiresAt),
  GET /wallet/topups/:id (статус для поллинга), POST /webhooks/topup (вебхук
  эквайринга: проверка подписи, идемпотентность по TopUp.externalId, зачисление
  ledger+balance в одной транзакции). Крипто-эквайринг — за интерфейсом
  PaymentProvider с sandbox/fake-реализацией (реальный провайдер подключим позже);
  обработка просроченных pending (status=expired).
- apps/web: экран /wallet по прототипу — карточка баланса, форма пополнения
  (сумма, актив/сеть), после создания — QR/адрес/таймер + поллинг статуса,
  успех-flash при зачислении; история транзакций. i18n EN/RU, loading/empty/error.
- Тесты: unit ledger-сервиса (зачисление, двойная запись, идемпотентность повторного
  вебхука, сходимость balanceAfter), smoke: создать пополнение → вебхук → баланс
  вырос → повторный вебхук ничего не меняет (фейки apps/api/src/testing/fakes.ts).

КРИТЕРИЙ ПРИЁМКИ: тестовое пополнение доходит до баланса корректно и идемпотентно;
повторный вебхук/повторный POST с тем же Idempotency-Key не задваивают; история и
баланс видны в ЛК; lint/typecheck/тесты зелёные, CI зелёный. DoD — CLAUDE.md/docs/16 §8.

КОНВЕНЦИИ (из CLAUDE.md): контракты вперёд; вертикальный срез фронт→API→БД→тесты;
деньги — ТОЛЬКО Decimal/строки Money + транзакции БД + ledger + идемпотентность,
никаких float; не логировать секреты/подписи; дизайн по docs/design/* и прототипу
(SVG-иконки, без эмодзи); i18n через ключи.

РАБОТА С GIT: веди на feature-ветке для этого эпика, понятные коммиты, в конце —
commit + push.

ЕСЛИ ВОЗНИКНУТ РАЗВИЛКИ (выбор провайдера/библиотек, структура, спорные решения) —
задавай вопросы в режиме asking (AskUserQuestion), не угадывай.

В КОНЦЕ СЕССИИ: обнови docs/SESSION-LOG.md (запись о сделанном + статус E3=готово +
следующий эпик E4), при изменении контрактов обнови docs/backend/, затем commit+push.
Подготовь готовый промт для следующей сессии (E4) в docs/NEXT-SESSION-PROMPT.md.
```

## Порядок эпиков (быстрый reference)

E0 каркас → E1 auth → E2 каталог → E3 кошелёк/пополнение → E4 корзина/заказы/оплата →
E5 выдача из стока → E6 прогрев (модель+очередь) → E7 инвентарь прокси/Octo →
E8 полная админка/операторка → E9 поддержка/уведомления → E10 гарантии/замены/возвраты →
E11 полировка/безопасность/запуск. Детали и оценки — `docs/16-development-plan.md`.

## Советы для качественной «мега-сессии»
- Одна сессия = один эпик (или его чёткая часть). E8 (админка) — дели на под-сессии.
- Всегда начинай с чтения `SESSION-LOG.md` — не начинай «с нуля».
- Держи контракты (OpenAPI/Prisma) синхронными с кодом — это память проекта.
- Не раздувай сессию: доведи срез до приёмки, задокументируй, запушь, подготовь
  следующий промт. Непрерывность важнее объёма за раз.
