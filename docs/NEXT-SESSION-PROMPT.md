# Стартовые промты сессий

Здесь лежит **мастер-промт (самоориентирующийся)**, **готовый промт для ближайшей
сессии** и **универсальный шаблон**. Скопируй нужный в начало новой сессии.

---

## ⭐ МАСТЕР-ПРОМТ (самоориентирующийся, рекомендуется)

> Работает на любой ветке: сам находит проект и определяет текущий эпик. Копируй целиком.

```
Ты продолжаешь разработку проекта AdVault — премиальный маркетплейс цифровых
товаров (аккаунты Google Ads: готовые из стока и под заказ с прогревом операторами).

ШАГ 0 — НАЙДИ ПРОЕКТ И СОРИЕНТИРУЙСЯ (обязательно, до любых действий):
1. Проверь, что в рабочей папке есть проект — файлы CLAUDE.md и docs/16-development-plan.md.
2. Если их нет (папка пустая / не та ветка), выполни:
     git fetch --all
     git checkout claude/advault-e9-support-notifications-grpdmg   # актуальная база: E0…E8 + весь E9
   (в main пока только документация/прототипы — код эпиков туда ещё не влит;
   проверяй: рядом с docs/ должны быть apps/ и packages/).
3. Прочитай для контекста, строго в этом порядке:
     - CLAUDE.md            — что строим, где что лежит, конвенции, границы;
     - docs/SESSION-LOG.md  — что уже сделано и какой эпик следующий;
     - docs/16-development-plan.md — детальный план (эпики E0–E11, вехи M0–M5);
     - раздел плана по текущему эпику + релевантные доки
       (docs/03-tech-stack, docs/04-architecture, docs/design/*, docs/backend/*).
4. Определи текущий эпик из docs/SESSION-LOG.md (блок «Текущий статус»).
   Если ещё ничего не начато — это эпик E0 «Каркас монорепо».

ЦЕЛЬ СЕССИИ — довести до готовности текущий эпик (сейчас это E10 «Гарантии, замены,
возвраты»; E0…E9 готовы). См. готовый промт E10 ниже — там детальные задачи и развилки.
Кратко:
  - Контракты вперёд: docs/backend/{prisma-schema,openapi}.md — модель гарантии/заявки
    на замену/возврат (WarrantyClaim?) + маршруты; затем код;
  - apps/api: клиентская заявка на замену/возврат в пределах warrantyHours с аудитом;
    замена стокового товара (новый StockItem) / возврат средств через ledger (Decimal!);
  - apps/web: экран «гарантия/замена» в заказе/ЛК; статусы заявки; i18n EN/RU;
  - тесты: unit/smoke (окно гарантии, идемпотентность возврата, скоупинг на владельца).
  Критерий приёмки: замена/возврат проходят с аудитом; статусы корректны; деньги через
  ledger; RBAC-скоупинг; lint/typecheck/тесты/CI зелёные.

КОНВЕНЦИИ (из CLAUDE.md, обязательно):
  - Контракты вперёд: меняешь поведение — сначала обнови OpenAPI/Prisma в docs/backend.
  - Вертикальный срез: фронт→API→БД→тесты.
  - Дизайн только по docs/design/* и прототипу; иконки — SVG-спрайт, БЕЗ эмодзи; i18n EN/RU через ключи.
  - Деньги: decimal + транзакции + ledger + идемпотентность. Секреты: шифрование + аудит.
    RBAC на админ/операторских маршрутах; danger-confirm на необратимом.
  - Граница платформы: НЕ автоматизируем прогрев/действия в аккаунтах/обход детекта —
    только бизнес-логистика (каталог, заказы, инвентарь, очереди, трекинг, выдача, аудит).

GIT: создай feature-ветку под эпик от актуальной базы (напр. feat/E1-auth);
понятные коммиты; в конце — git push -u origin <ветка>.

ВОПРОСЫ: при любой значимой развилке (выбор библиотек, структура, спорное решение или
неоднозначность требований) — спрашивай в режиме asking (AskUserQuestion), не угадывай.

В КОНЦЕ СЕССИИ (ритуал):
  1. Обнови docs/SESSION-LOG.md: что сделано, статус эпика (E9=готово), следующий эпик (E10).
  2. Обнови статусы в docs/16 и контракты в docs/backend при изменениях.
  3. commit + push.
  4. Подготовь готовый промт следующего эпика в docs/NEXT-SESSION-PROMPT.md.
```

Для следующих эпиков просто замени в блоке «ЦЕЛЬ СЕССИИ» задачи и критерий на нужный
эпик из `docs/16` (мастер-промт сам подхватит текущий эпик из SESSION-LOG).

## ✅ ГОТОВЫЙ ПРОМТ — Сессия E10 (Гарантии, замены, возвраты)

> Скопируй блок ниже целиком в новую сессию.

```
Продолжаем разработку проекта AdVault (маркетплейс цифровых товаров: аккаунты Google
Ads — готовые из стока и под заказ с прогревом операторами). Готовы E0…E7 (каркас, auth,
каталог, кошелёк, заказы/оплата, выдача из стока, прогрев, инвентарь прокси/Octo),
ВЕСЬ E8 (полная админка/операторка) и ВЕСЬ E9 (Поддержка и уведомления: клиентский
портал тикетов `/tickets*` scoped на владельца с скрытыми internal-заметками; in-app
Notification+бейдж; email+in-app по order.paid/warming.ready/ticket.reply на per-locale
шаблонах Settings; админ-индикатор ответа покупателя + авто pending→open). Всё под
RBAC/аудитом, проверено вживую (curl полного цикла на реальном Postgres+Redis) и e2e.

ВАЖНО — БАЗА КОДА: актуальный код в ветке claude/advault-e9-support-notifications-grpdmg
(E0…E8 + ВЕСЬ E9; в main ещё не влито). Если в рабочей папке нет apps/ и packages/:
  git fetch --all
  git checkout claude/advault-e9-support-notifications-grpdmg
Новую feature-ветку для E10 создавай от неё.

ПЕРЕД РАБОТОЙ прочитай: CLAUDE.md (контекст/конвенции/границы — провижининг ВРУЧНУЮ,
платформа фиксирует связки), docs/SESSION-LOG.md (верхняя запись «E9» — что сделано и
долги), docs/16-development-plan.md §E10, docs/14-order-lifecycle.md (статусы заказа/
позиции: refunded/replaced), docs/11 (типы товаров, warrantyHours), docs/09 (безопасность/
границы), docs/backend/{prisma-schema,openapi}.md. Уже есть: `ProductVariant.warrantyHours`,
ручной refund в админ-финансах (E8), выдача из стока (E5)/прогрев (E6), ledger (E3, Decimal).

ЦЕЛЬ СЕССИИ — E10 «Гарантии, замены, возвраты» (v1.1):
- Клиентская заявка на замену/возврат по доставленной позиции в пределах окна гарантии
  (warrantyHours от момента выдачи); статусы заявки (requested→approved/rejected→
  replaced/refunded), привязка к OrderItem/Delivery, аудит каждого перехода.
- Замена: для READY_STOCK — выдать новый StockItem из пула (резерв→продажа, новый Delivery,
  deliveryStatus=replaced); для MADE_TO_ORDER — новая warm-задача/ручная замена оператором.
- Возврат: возврат средств через LedgerService (credit, Decimal, идемпотентность), статус
  позиции/заказа refunded (агрегат по docs/14).
- Админ-сторона: очередь заявок (approve/reject/выполнить), RBAC (FINANCE_STAFF/ELEVATED
  на деньги), danger-confirm на необратимом.
Начни с asking-развилки: модель заявки (новая WarrantyClaim vs расширение Order/Delivery);
кто одобряет (авто-политика по причине vs всегда ручное); политика окна (жёсткое
warrantyHours vs с grace). Контракты вперёд (docs/backend/*), затем код; вертикальный
срез фронт→API→БД→тесты.

КОНВЕНЦИИ (CLAUDE.md): деньги — ТОЛЬКО Decimal+транзакции+ledger+идемпотентность (возврат
= credit-запись, никаких float); секреты — шифрование+доступ по праву+аудит, НЕ логировать;
RBAC (группы в apps/api/src/auth/roles.ts) — клиентские заявки строго scoped на владельца;
замена стока — тот же двухфазный резерв, что в E5; дизайн по docs/design/* и прототипу
(SVG-иконки из спрайта, без эмодзи); i18n EN/RU через ключи (EN/RU в синхроне — тест
locales.spec); состояния loading/empty/error; ГРАНИЦА ПЛАТФОРМЫ — только бизнес-логистика.

РАЗВИЛКИ (asking, не угадывай): модель заявки; политика одобрения; окно гарантии/grace;
частичные возвраты по позициям vs весь заказ.

КРИТЕРИЙ ПРИЁМКИ: покупатель оформляет замену/возврат по доставленной позиции в окне
гарантии end-to-end; замена выдаёт новый актив, возврат зачисляет средства через ledger
(идемпотентно); статусы позиции/заказа корректны (docs/14); аудит каждого шага; RBAC-
скоупинг; lint/typecheck/тесты/build зелёные, CI зелёный. DoD — docs/16 §8.

В КОНЦЕ: обнови docs/SESSION-LOG.md (запись + статус E10/переход к E11 + следующий шаг),
при изменении контрактов/схемы обнови docs/backend/ (+миграция, если нужна), commit+push,
подготовь промт следующей сессии (E11 — Полировка/безопасность/запуск) в
docs/NEXT-SESSION-PROMPT.md.
```

## Порядок эпиков (быстрый reference)

E0 каркас → E1 auth → E2 каталог → E3 кошелёк/пополнение → E4 корзина/заказы/оплата →
E5 выдача из стока → E6 прогрев (модель+очередь) → E7 инвентарь прокси/Octo →
E8 полная админка/операторка → E9 поддержка/уведомления → E10 гарантии/замены/возвраты →
E11 полировка/безопасность/запуск. Детали и оценки — `docs/16-development-plan.md`.

## Советы для качественной «мега-сессии»
- Одна сессия = один эпик (или его чёткая часть). E8 (админка) — дели на под-сессии.
- Всегда начинай с чтения `SESSION-LOG.md` — не начинай «с нуля».
- Держи контракты (OpenAPI/Prisma) синхронными с кодом — это память проекта.
- Не раздувай сессию: доведи срез до приёмки, задокументируй, запушь, подготовь
  следующий промт. Непрерывность важнее объёма за раз.
