# Стартовые промты сессий

Здесь лежит **мастер-промт (самоориентирующийся)**, **готовый промт для ближайшей
сессии** и **универсальный шаблон**. Скопируй нужный в начало новой сессии.

---

## ⭐ МАСТЕР-ПРОМТ (самоориентирующийся, рекомендуется)

> Работает на любой ветке: сам находит проект и определяет текущий эпик. Копируй целиком.

```
Ты продолжаешь разработку проекта AdVault — премиальный маркетплейс цифровых
товаров (аккаунты Google Ads: готовые из стока и под заказ с прогревом операторами).

ШАГ 0 — НАЙДИ ПРОЕКТ И СОРИЕНТИРУЙСЯ (обязательно, до любых действий):
1. Проверь, что в рабочей папке есть проект — файлы CLAUDE.md и docs/16-development-plan.md.
2. Если их нет (папка пустая / не та ветка), выполни:
     git fetch --all
     git checkout claude/advault-e8-admin-completion-c5l1u4   # актуальная база: E0…E7 + весь E8
   (в main пока только документация/прототипы — код эпиков туда ещё не влит;
   проверяй: рядом с docs/ должны быть apps/ и packages/).
3. Прочитай для контекста, строго в этом порядке:
     - CLAUDE.md            — что строим, где что лежит, конвенции, границы;
     - docs/SESSION-LOG.md  — что уже сделано и какой эпик следующий;
     - docs/16-development-plan.md — детальный план (эпики E0–E11, вехи M0–M5);
     - раздел плана по текущему эпику + релевантные доки
       (docs/03-tech-stack, docs/04-architecture, docs/design/*, docs/backend/*).
4. Определи текущий эпик из docs/SESSION-LOG.md (блок «Текущий статус»).
   Если ещё ничего не начато — это эпик E0 «Каркас монорепо».

ЦЕЛЬ СЕССИИ — довести до готовности текущий эпик (сейчас это E9 «Поддержка и уведомления»;
E0…E8 готовы). См. готовый промт E9 ниже — там детальные задачи и развилки. Кратко:
  - Контракты вперёд: docs/backend/{prisma-schema,openapi}.md — клиентские маршруты
    тикетов `/tickets*` (scoped на владельца) + Notification (если вводим модель); затем код;
  - apps/api: клиентский портал тикетов поверх Ticket/TicketMessage (internal-сообщения
    скрыты от покупателя!); уведомления email+in-app по событиям на шаблонах из Settings;
  - apps/web: экраны тикетов в ЛК + центр уведомлений/бейдж; i18n EN/RU;
  - тесты: unit/smoke сценариев (скоупинг тикетов, доставка уведомлений).
  Критерий приёмки: покупатель ведёт тикет из ЛК end-to-end (internal скрыты); уведомления
  доставляются по событиям; RBAC-скоупинг; lint/typecheck/тесты/CI зелёные.

КОНВЕНЦИИ (из CLAUDE.md, обязательно):
  - Контракты вперёд: меняешь поведение — сначала обнови OpenAPI/Prisma в docs/backend.
  - Вертикальный срез: фронт→API→БД→тесты.
  - Дизайн только по docs/design/* и прототипу; иконки — SVG-спрайт, БЕЗ эмодзи; i18n EN/RU через ключи.
  - Деньги: decimal + транзакции + ledger + идемпотентность. Секреты: шифрование + аудит.
    RBAC на админ/операторских маршрутах; danger-confirm на необратимом.
  - Граница платформы: НЕ автоматизируем прогрев/действия в аккаунтах/обход детекта —
    только бизнес-логистика (каталог, заказы, инвентарь, очереди, трекинг, выдача, аудит).

GIT: создай feature-ветку под эпик от актуальной базы (напр. feat/E1-auth);
понятные коммиты; в конце — git push -u origin <ветка>.

ВОПРОСЫ: при любой значимой развилке (выбор библиотек, структура, спорное решение или
неоднозначность требований) — спрашивай в режиме asking (AskUserQuestion), не угадывай.

В КОНЦЕ СЕССИИ (ритуал):
  1. Обнови docs/SESSION-LOG.md: что сделано, статус эпика (E8=готово), следующий эпик (E9).
  2. Обнови статусы в docs/16 и контракты в docs/backend при изменениях.
  3. commit + push.
  4. Подготовь готовый промт следующего эпика в docs/NEXT-SESSION-PROMPT.md.
```

Для следующих эпиков просто замени в блоке «ЦЕЛЬ СЕССИИ» задачи и критерий на нужный
эпик из `docs/16` (мастер-промт сам подхватит текущий эпик из SESSION-LOG).

## ✅ ГОТОВЫЙ ПРОМТ — Сессия E9 (Поддержка и уведомления)

> Скопируй блок ниже целиком в новую сессию.

```
Продолжаем разработку проекта AdVault (маркетплейс цифровых товаров: аккаунты Google
Ads — готовые из стока и под заказ с прогревом операторами). Готовы E0…E7 (каркас, auth,
каталог, кошелёк, заказы/оплата, выдача из стока, прогрев, инвентарь прокси/Octo) и
ВЕСЬ E8 (полная админка/операторка): роли operator/manager/admin; Orders + Warming
Kanban/operator-workspace + Inventory-UI + Stock read; Finance (refund+ручная выдача+
сверка ledger) + Users + Promo; Catalog & Bundles CRUD + Warming plans CRUD с
версионированием; **Dashboard/Reports (агрегаты §1/§14), Tickets (Ticket/TicketMessage),
Staff&roles UI, Settings (key-value)**. Всё под RBAC/аудитом, проверено вживую под ролями
(curl полного цикла на реальном Postgres+Redis).

ВАЖНО — БАЗА КОДА: актуальный код в ветке claude/advault-e8-admin-completion-c5l1u4
(E0…E7 + ВЕСЬ E8; в main ещё не влито). Если в рабочей папке нет apps/ и packages/:
  git fetch --all
  git checkout claude/advault-e8-admin-completion-c5l1u4
Новую feature-ветку для E9 создавай от неё.

ПЕРЕД РАБОТОЙ прочитай: CLAUDE.md (контекст/конвенции/границы — провижининг ВРУЧНУЮ,
платформа фиксирует связки), docs/SESSION-LOG.md (верхняя запись «E8, часть 4» — что
сделано и долги), docs/16-development-plan.md §E9, docs/13-admin-panel-spec.md §13 (Tickets),
docs/09 (безопасность/границы), docs/backend/{prisma-schema,openapi}.md. Уже есть:
модели Ticket/TicketMessage (админ-сторона тикетов) и Settings с шаблонами уведомлений
(orderPaid/warmingReady/ticketReply) — E9 строится ПОВЕРХ них; mailer в apps/api/src/mailer
пока заглушка.

ЦЕЛЬ СЕССИИ — E9 «Поддержка и уведомления»:
- Клиентский портал тикетов: покупатель создаёт тикет из ЛК (привязка к своему заказу),
  ведёт переписку, видит только НЕ-internal сообщения; реюз моделей Ticket/TicketMessage
  (нужны клиентские маршруты `/tickets*` под RequireAuth, отдельные от админских).
- Уведомления: транзакционные email + in-app по событиям (order.paid, warming.ready,
  ticket.reply) на шаблонах из Settings (i18n EN/RU, подстановка {{number}} и т.п.);
  доставка через mailer-интерфейс (sandbox/заглушка) + возможно очередь (BullMQ уже в стеке).
  In-app — модель Notification (read/unread) + бейдж в шапке.
- Админ-сторона: индикатор новых клиентских ответов в очереди тикетов; авто-переход
  статуса тикета при ответе покупателя (pending→open).
Начни с asking-развилки: объём (портал+email+in-app vs подмножество); модель Notification
(отдельная таблица vs только email); движок доставки (синхронно через mailer vs очередь BullMQ).
Контракты вперёд (docs/backend/*), затем код; вертикальный срез фронт→API→БД→тесты.

КОНВЕНЦИИ (CLAUDE.md): деньги — ТОЛЬКО Decimal+транзакции+ledger+идемпотентность; секреты —
шифрование+доступ по праву+аудит, НЕ логировать; RBAC (группы в apps/api/src/auth/roles.ts:
STAFF/ORDERS_STAFF/ELEVATED/ADMIN_ONLY/CATALOG_STAFF/SUPPORT_STAFF/REPORTS_STAFF) —
клиентские тикеты строго scoped на владельца (покупатель видит только свои тикеты и только
не-internal сообщения!); дизайн по docs/design/* и прототипу (SVG-иконки из спрайта, без
эмодзи); i18n EN/RU через ключи (EN/RU в синхроне — тест locales.spec); состояния
loading/empty/error; ГРАНИЦА ПЛАТФОРМЫ — только бизнес-логистика.

РАЗВИЛКИ (asking, не угадывай): объём E9; модель Notification (таблица vs только email);
движок доставки (mailer-синхронно vs BullMQ-очередь); нужен ли WebSocket для in-app или
поллинг достаточно.

КРИТЕРИЙ ПРИЁМКИ: покупатель ведёт тикет из ЛК end-to-end (создание→переписка, internal-
заметки СКРЫТЫ); уведомления доставляются по событиям на шаблонах из Settings; in-app
бейдж работает; RBAC-скоупинг тикетов на владельца; lint/typecheck/тесты/build зелёные,
CI зелёный. DoD — docs/16 §8.

В КОНЦЕ: обнови docs/SESSION-LOG.md (запись + статус E9/переход к E10 + следующий шаг), при
изменении контрактов/схемы обнови docs/backend/ (+миграция для Notification, если нужна),
commit+push, подготовь промт следующей сессии (E10 — Гарантии/замены/возвраты) в
docs/NEXT-SESSION-PROMPT.md.
```

## Порядок эпиков (быстрый reference)

E0 каркас → E1 auth → E2 каталог → E3 кошелёк/пополнение → E4 корзина/заказы/оплата →
E5 выдача из стока → E6 прогрев (модель+очередь) → E7 инвентарь прокси/Octo →
E8 полная админка/операторка → E9 поддержка/уведомления → E10 гарантии/замены/возвраты →
E11 полировка/безопасность/запуск. Детали и оценки — `docs/16-development-plan.md`.

## Советы для качественной «мега-сессии»
- Одна сессия = один эпик (или его чёткая часть). E8 (админка) — дели на под-сессии.
- Всегда начинай с чтения `SESSION-LOG.md` — не начинай «с нуля».
- Держи контракты (OpenAPI/Prisma) синхронными с кодом — это память проекта.
- Не раздувай сессию: доведи срез до приёмки, задокументируй, запушь, подготовь
  следующий промт. Непрерывность важнее объёма за раз.
