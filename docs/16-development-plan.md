# 16 — Детальный план разработки

Полный план создания AdVault с учётом расширенной модели (готовые + прогрев,
инвентарь прокси/Octo, полная админка/операционка). Заменяет и расширяет
краткую дорожную карту [10](./10-roadmap-sessions.md).

> Оценки — индикативные (dev-days для 1–2 разработчиков), для калибровки объёма, не
> обязательства. Порядок эпиков учитывает зависимости.

## 1. Подход и принципы

- **Итеративно, по вехам**: каждая веха оставляет систему в рабочем, демонстрируемом
  состоянии. Внутри вехи — эпики, внутри эпиков — задачи с критериями приёмки.
- **Вертикальные срезы**: фича проходит фронт→API→БД→тесты, а не «сначала весь бэк».
- **Контракты вперёд**: API (OpenAPI) и схема БД (Prisma) фиксируются до массового
  кодинга фичи (заготовки уже есть в `docs/backend/`).
- **Качество на входе**: линт/типы/тесты/ревью в CI с первого дня.
- **Безопасность и деньги — под особым контролем**: транзакции, ledger, идемпотентность,
  шифрование, аудит — не «потом», а в соответствующих эпиках.

## 2. Команда (рекомендация)

| Роль | Зона |
|------|------|
| Fullstack lead | архитектура, ревью, сложные срезы |
| Frontend dev | витрина, ЛК, админка, дизайн-система |
| Backend dev | API, БД, платежи, воркеры, интеграции |
| QA / автотесты | e2e ключевых потоков, регресс |
| (part-time) DevOps | инфра, CI/CD, мониторинг, секреты |
| (part-time) Design/PM | приоритеты, UX-ревью, тексты |

Минимально проект тянут 1–2 сильных fullstack + QA.

## 3. Инфраструктура и окружения (до фич)

- Монорепо (pnpm workspaces): `apps/web`, `apps/api`, `apps/admin` (или общий web с
  разделами), `packages/{types,ui,config}`.
- Окружения: `dev` (docker-compose), `staging`, `prod`. Разделение секретов.
- CI (GitHub Actions): lint + typecheck + unit + build; на PR — e2e-smoke; на main —
  деплой staging.
- Наблюдаемость: Sentry, структурные логи, базовые метрики; алерты на платежи/выдачи.
- Секреты — в менеджере секретов/KMS (ключи шифрования payload).

## 4. Эпики (детально)

Легенда приоритета: **MVP** / **v1.1** / **v2**. Оценка — примерные dev-days.

### E0. Каркас и платформа (MVP) — ~6–9 dd — ✅ готово
- Монорепо, tooling (ESLint/Prettier/tsconfig), docker-compose (postgres, redis, api, web).
- NestJS-скелет: health, конфиг, подключение Postgres(Prisma)+Redis, Swagger.
- React(Vite)+Tailwind+роутинг+i18n(EN/RU)+дизайн-токены из прототипа.
- CI зелёный.
- **Приёмка**: локально поднимается; фронт видит `/health`; сборка проходит в CI.
- **Зависимости**: нет.

### E1. Аутентификация и аккаунты (MVP) — ~6–8 dd — ✅ готово
- Prisma: User, роли, статусы. Регистрация/вход, JWT (access+refresh, ротация),
  подтверждение email, восстановление пароля, rate-limit на auth.
- Фронт: экраны из прототипа `auth.html` (login/register/forgot/reset/verify),
  guard-роуты, хранение токенов. Профиль `/me`, локаль.
- **Приёмка**: полный цикл регистрации→входа→ЛК; ошибки/лимиты работают.
- **Зависимости**: E0.

### E2. Каталог и продуктовая модель (MVP) — ~8–11 dd — ✅ готово
- Prisma: Category, Product, ProductVariant (+ fulfillmentType, goal, tier, bundleSpec,
  warrantyHours), *Translation.
- API: категории, список/карточка, фильтры, поиск, наличие/ETA.
- Фронт: витрина, каталог с фильтрами, карточка товара с выбором варианта, показом
  комплекта и ETA (для warm). i18n-контент.
- Сидер демо-данных (в т.ч. warm-варианты: google_ads, chrome_extension_dev).
- **Приёмка**: каталог просматривается; видно тип выдачи, комплект, ETA; фильтры/поиск.
- **Зависимости**: E0 (E1 желателен для «купить»).

### E3. Кошелёк и пополнение криптой (MVP) — ~7–10 dd — ✅ готово
- Prisma: LedgerEntry, TopUp. Интеграция крипто-эквайринга (sandbox): создание
  пополнения, вебхук с проверкой подписи, идемпотентность, ledger+баланс в транзакции.
- Фронт: экран пополнения (QR/адрес/таймер/статус), история транзакций.
- **Приёмка**: тестовое пополнение доходит до баланса корректно и идемпотентно.
- **Зависимости**: E1.

### E4. Корзина, заказы, оплата с баланса (MVP) — ~7–10 dd — ✅ готово
- Prisma: Order, OrderItem (deliveryStatus, тип). Корзина, checkout: проверка наличия,
  резерв стока, списание с баланса (ledger+транзакция), промокоды (базово).
- Фронт: экраны из `checkout.html` (корзина, промокод, степпер, оплата, успех-flash).
- **Приёмка**: покупка стокового товара за баланс; заказ создаётся/оплачивается.
- **Зависимости**: E2, E3.

### E5. Выдача из стока (READY_STOCK) (MVP) — ~6–8 dd — ✅ готово
- Prisma: StockItem (payloadHash-дедуп), Delivery, AuditLog; payload шифруется
  AES-256-GCM (версионируемый ключ из env). Двухфазный резерв (available→reserved +
  reservedUntil + Redis TTL, sweep просроченных), авто-выдача после оплаты
  (reserved→sold + Delivery + deliveryStatus=delivered, статус заказа — агрегат по
  docs/14; stockCount пересчитывается от пула); импорт стоков (JSON/CSV/TXT) в админке
  (RBAC admin, отчёт added/skipped). `GET /orders/:id/items/:itemId/delivery` —
  расшифровка только владельцу + запись в AuditLog.
- Фронт: Vault в деталях заказа (маскирование → «Показать», копирование, скачивание
  .txt, micro-flash), i18n EN/RU, loading/empty/error.
- **Приёмка**: авто-выдача end-to-end; данные доступны только владельцу (аудит);
  импорт работает — ✅ проверено (curl + Chromium/Playwright).
- **Зависимости**: E4.

### E6. Прогрев: модель и очередь (MADE_TO_ORDER) (MVP-core) — ~12–16 dd — ✅ готово
- Prisma: WarmingPlan, WarmingStageTemplate, WarmingJob, WarmingTask, AccountAsset,
  Bundle, BundleComponent (см. [15](./15-data-model-extensions.md)).
- Логика: при оплате warm-позиции — создание Job (queued), расчёт ETA из плана; смена
  статусов (queued→assigned→in_progress→qc→ready→delivered), пересчёт ETA при on_hold.
- API: очередь, назначение, переходы этапов, сборка/QC/выдача Bundle.
- Фронт (покупатель): статус заказа с ETA и прогрессом по этапам; уведомления.
- **Приёмка**: warm-заказ проходит весь цикл (через админ-действия) до выдачи в Vault.
- **Зависимости**: E4, E5 (Delivery/шифрование).

### E7. Инвентарь: прокси и Octo-профили (MVP-core) — ~7–10 dd — ✅ готово
- Prisma: ProxyItem, OctoProfile. Админ-CRUD, импорт, статусы, привязка к Job.
- Логика: резерв/привязка ресурсов из задачи; шифрование credentials/exportRef.
- **Приёмка**: оператор привязывает прокси и профиль к warm-заказу; попадают в Bundle.
- **Зависимости**: E6.

### E8. Полная админка / операторка (MVP-core) — ~16–22 dd — ✅ готово
> **Сделано (часть 1):** RBAC-роли operator/manager (аддитивно на User.role), admin
> Orders (таблица+деталь), Warming Kanban + operator workspace (assign/переходы/чек-лист/
> захват аккаунта/resolve + bind прокси/Octo из E7), Inventory-UI (прокси/Octo:
> список/создание/импорт), admin Stock read.
> **Сделано (часть 2):** Finance — ручной refund (идемпотентный ledger-credit per-orderItem,
> warm-job→refunded, сверка ledger) + ручная выдача (шифрование payload); Users —

> (admin-only) с аудитом; Promo CRUD. Всё под RBAC (FINANCE_STAFF/ELEVATED/ADMIN_ONLY),
> danger-confirm + AuditLog. Проверено вживую end-to-end под ролями admin/support/user.
> **Сделано (часть 3):** Catalog & Bundles CRUD — категории/товары/варианты (SKU) с

> (manager+), AuditLog на каждую мутацию. Проверено вживую (Postgres+Redis): цикл
> план→товар→вариант→публикация→витрина + версионирование с пересчётом ETA.
> **Сделано (часть 4 — финал E8):** Dashboard + Reports/Analytics (read-only агрегаты

> (модели Ticket/TicketMessage + миграция; очередь/деталь/create-от-лица-покупателя/
> reply+internal-note/assign/status; RBAC support+; аудит); Staff&roles UI (список
> сотрудников с живой загрузкой, смена роли admin-only); Settings (key-value стор,
> магазин/языки/шаблоны уведомлений; интеграции — read-only флаги из env, без секретов;
> RBAC admin). Проверено вживую (Postgres+Redis) под ролями end-to-end.
> **Долг (в E9+):** inline-edit промо, attachments/макросы тикетов, real-time очередей,
> reports-джойны в памяти → raw-SQL при росте объёмов.
- RBAC (Owner/Admin/Manager/Operator/Support), StaffUser, гварды, аудит-лог.
- Модули (см. [13](./13-admin-panel-spec.md)): Dashboard, Orders, **Warming Kanban**,

### E10. Гарантии, замены, возвраты (v1.1) — ~5–7 dd ✅ ГОТОВО
- Prisma: **WarrantyClaim** (+enum WarrantyClaimType/Status; NotificationType +3).

## 6. Разбивка по рабочим сессиям

Каждая сессия = 1–2 эпика (или их часть), заканчивается коммитом/пушем и обновлением
статуса. Рекомендуемый порядок: **E0 → E1 → E2 → E3 → E4 → E5 → E6 → E7 → E8 → E9 →
E10 → E11**. E8 (админка) крупный — дели на под-сессии по модулям (сначала Orders +
Warming Kanban + Operator workspace + инвентарь, затем Products/Plans/Finance/Reports).

## 7. Сквозные требования (во всех эпиках)

- **i18n** EN/RU (ключи, без хардкода), форматирование по локали.
- **Безопасность**: RBAC, валидация входов, rate-limit, шифрование секретов,
  идемпотентность денег/выдач, аудит опасных действий.
- **Тестирование**: unit (сервисы/логика денег и статусов), e2e (ключевые потоки),
  a11y-проверки; тесты — часть Definition of Done.
- **Наблюдаемость**: логи/ошибки/метрики; алерты на платежи, выдачи, SLA-просрочки.
- **Дизайн-система**: единые токены/компоненты/иконки (см. `docs/design/*`, прототип).
- **Производительность**: кэш каталога, скелетоны, ленивые загрузки, LCP<2.5s.

## 8. Definition of Done (для задачи/эпика)
- Код проходит lint/typecheck/тесты; покрыты ключевые ветки.
- Реализованы состояния UI (loading/empty/error) и a11y-минимум.
- Тексты EN/RU через i18n; опасные действия — подтверждение+аудит.
- Обновлена документация/OpenAPI/Prisma при изменении контрактов.
- Пройдено ревью; фича продемонстрирована на staging.

## 9. Реестр рисков

| Риск | Влияние | Митигейшн |
|------|---------|-----------|
| Платёжный провайдер/хостинг отклоняют тематику | высокое | заранее выбрать допускающих; иметь запасных; юр-документы |
| Правовые/ToS-риски (прогрев, Octo, прокси) | высокое | юр-консультация, ToS/Refund/Privacy, лимиты, см. [09] |
| Узкое место — ручной прогрев (скорость/масштаб) | среднее | удобная операторка, SLA, распределение, позже автоматизация (v2) |
| Утечка секретов (аккаунты/прокси/профили) | высокое | шифрование, доступ по праву, аудит, минимизация |
| Двойная выдача/двойное списание | высокое | транзакции, идемпотентность, резерв стока, тесты |
| Срыв ETA, недовольство | среднее | честные оценки, эскалация, уведомления, политика возвратов |
| Раздувание объёма админки (E8) | среднее | резать на под-модули, приоритизировать операторский критпуть |
