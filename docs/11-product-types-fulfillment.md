# 11 — Типы товаров и модели выдачи

Расширение продуктовой модели: помимо готовых аккаунтов из стока появляются
**аккаунты под заказ с прогревом** под конкретную цель, с гибким комплектом выдачи
(аккаунт / прокси / Octo-профиль). Этот документ — источник истины по продуктовой
таксономии; операции прогрева детализированы в [12](./12-warming-pipeline.md),
жизненный цикл заказа — в [14](./14-order-lifecycle.md).

## Две модели выдачи (fulfillment)

| Модель | Код | Как работает | ETA |
|--------|-----|--------------|-----|
| Готовый из стока | `READY_STOCK` | Аккаунт уже подготовлен, лежит в стоке. Выдача мгновенная после оплаты. | секунды |
| Под заказ с прогревом | `MADE_TO_ORDER` | После оплаты заказ уходит в очередь прогрева; оператор готовит/прогревает аккаунт под цель и собирает комплект. | часы–дни (по плану) |

Модель задаётся на уровне **варианта товара** (`ProductVariant.fulfillmentType`).
Один товар может иметь и стоковые, и заказные варианты (например, «Google Ads — из
стока» и «Google Ads — прогрев 7 дней»).

## Продуктовая таксономия

```
Category (раздел)  →  Product (товар)  →  ProductVariant (SKU)  →  Bundle (комплект)
     |                     |                      |                        |
  Google Ads           US Verified          Standard/Aged/Agency     account [+proxy] [+octo] [+guide]
  Dev accounts         Chrome extensions    Warm 7d / Warm 14d       account [+recovery] [+guide]
  Proxies              Residential US       Monthly                  proxy
  Anti-detect          Octo profiles        Per profile              octo_profile
```

### Цели прогрева (goal) — примеры, список расширяемый
- **`google_ads`** — прогретый рекламный аккаунт Google Ads. Комплект обычно:
  аккаунт + выделенный прокси + Octo-профиль + гайд.
- **`chrome_extension_dev`** — Google-аккаунт разработчика для публикации расширений
  в Chrome Web Store. Комплект: аккаунт + recovery-данные + гайд (прокси/профиль — опц.).
- **`gmail_aged`**, **`youtube`**, **`merchant`**, … — добавляются как новые goal-значения.

> Цель (`goal`) определяет, какой **план прогрева** применяется и какой **комплект**
> собирается. Оба — конфигурируются на товаре/варианте, не хардкодятся.

## Комплект выдачи (Bundle) — конфигурируется по товару

Комплект — набор **компонентов**, включённых в вариант. Возможные компоненты:

| Компонент | Код | Что это |
|-----------|-----|---------|
| Аккаунт | `ACCOUNT` | Логин/пароль и связанные данные (обязателен) |
| Прокси | `PROXY` | Выделенный прокси, привязанный к аккаунту (тип/гео по товару) |
| Octo-профиль | `OCTO_PROFILE` | Профиль антидетект-браузера (экспорт/шеринг) под аккаунт+прокси |
| Recovery | `RECOVERY` | Резервная почта / данные восстановления |
| 2FA/секреты | `SECRETS` | TOTP-seed, cookies и т.п. (если применимо) |
| Гайд | `GUIDE` | Инструкция по входу и использованию |
| Гарантия | `WARRANTY` | Окно замены (например, 24–72 ч) |

Каждый вариант задаёт **список включённых компонентов** и их параметры (тип прокси,
гео, срок прокси, тип профиля). Отсюда «Настраивается по товару»: где-то только
`ACCOUNT`, где-то полный `ACCOUNT+PROXY+OCTO_PROFILE+GUIDE`.

## Клиентский сценарий заказа с прогревом (POV покупателя)

```
1. Покупатель выбирает товар с fulfillment = MADE_TO_ORDER, цель (goal), опции
   (гео, тариф прогрева, состав комплекта — если варианты).
2. Видит на карточке: ориентировочное время выдачи (ETA), состав комплекта, гарантию.
3. Оплачивает сразу (баланс или крипта).
4. Заказ создаётся со статусом «Queued» и показывает:
   - оценку времени до выдачи (ETA-таймер),
   - текущий статус/этап («Ваш аккаунт готовится», прогресс по этапам),
   - историю статусов.
5. По готовности статус → «Delivered», комплект появляется в разделе Vault
   (аккаунт + прокси + Octo-профиль + гайд), приходит уведомление (in-app + email).
```

Ключевое: **оплата сразу, ETA и живой статус**. Покупатель всегда видит, на каком
этапе его заказ и сколько примерно ждать.

## Провижининг (POV платформы) — операторы вручную, система трекает

- После оплаты MADE_TO_ORDER-заказ попадает в **очередь прогрева** (по goal/скиллу).
- **Оператор** берёт задачу, идёт по **плану прогрева** (этапы-чек-лист), отмечает
  прогресс, фиксирует использованные ресурсы: какой аккаунт, какой прокси из инвентаря,
  какой Octo-профиль.
- По завершении собирает **комплект выдачи**, проходит QC, помечает «Ready» → «Delivered».
- Всё выполняется людьми; платформа — это **система учёта, задач, статусов и SLA**
  (см. [12](./12-warming-pipeline.md) и [13](./13-admin-panel-spec.md)). Платформа не
  автоматизирует сам прогрев — она им управляет и его отслеживает.

## Ценообразование и опции

- Стоковые варианты — фиксированная цена, мгновенно.
- Заказные варианты — цена по тарифу прогрева (дольше/качественнее = дороже),
  надбавки за компоненты комплекта (прокси, профиль), за гео/срочность.
- Опции варианта (гео, тариф, состав) влияют на итоговую цену и на ETA.
