# 14 — Жизненный цикл заказа и выдачи (state machines)

Точная модель статусов и переходов для обоих типов выдачи. Служит контрактом между
фронтом, API, воркерами и админкой.

## Статусы заказа (Order.status)

```
pending ─► paid ─┬─► fulfilling ─► partially_delivered ─► delivered
                 │                        │
                 ├─► delivered (если все позиции — стоковые и выданы сразу)
                 └─► cancelled / refunded (по политике)
```

Заказ может содержать позиции обоих типов; статус заказа — агрегат статусов позиций.

## Статусы позиции (OrderItem.deliveryStatus)

### READY_STOCK (мгновенно)
```
pending ─► reserved ─► delivered
                 └─► out_of_stock ─► (перевод в manual/warm | возврат)
```

### MADE_TO_ORDER (с прогревом) — привязан к Warming Job
```
pending ─► queued ─► assigned ─► in_progress ─► qc ─► ready ─► delivered
                          │                       │
                          ├─► on_hold             └─► rework ─► in_progress
                          └─► failed ─► reassigned | refunded
```

## Warming Job — сопоставление со статусами позиции

| Job stage | OrderItem.deliveryStatus | Что видит покупатель |
|-----------|--------------------------|----------------------|
| Created/Queued | queued | «В очереди. ETA ~ N» |
| Assigned | assigned | «Взято в работу» |
| In progress (этапы) | in_progress | «Готовится: этап k из N», прогресс |
| QC | qc | «Финальная проверка» |
| Ready | ready | «Почти готово» |
| Delivered | delivered | «Готово! Комплект в Vault» |
| On hold | on_hold | «Небольшая задержка, новая оценка ~ M» |
| Failed/Rework | in_progress/refunded | «Дорабатываем» / возврат |

## ETA и таймеры

- **ETA (покупателю)** рассчитывается при оплате: сумма ожидаемых длительностей
  этапов плана прогрева (+ буфер очереди при высокой загрузке).
- **SLA (внутренний)**: дедлайн на этап/задачу; при приближении/просрочке —
  подсветка и эскалация менеджеру.
- При переходе в `on_hold` ETA пересчитывается и покупатель уведомляется.

## Оплата и деньги (сцепка со статусами)

- **paid**: списание с баланса / подтверждение крипто-оплаты (ledger-запись).
- **refunded**: возврат на внутренний баланс (ledger-credit), причина + аудит.
- Для warm-заказов деньги списываются сразу при оплате; при `failed`/невозможности
  выдать в срок — авто/ручной возврат по политике.
- Идемпотентность оплат и выдач (Idempotency-Key, уникальные externalId).

## Уведомления (события → каналы)

| Событие | Канал |
|---------|-------|
| Оплата принята | in-app + email |
| Warm: взято в работу / смена этапа | in-app (email на ключевых) |
| Задержка (on_hold), новая ETA | in-app + email |
| Ready / Delivered | in-app + email |
| Замена одобрена/выполнена | in-app + email |
| Возврат выполнен | in-app + email |

## Edge-cases и политики

- **Сток кончился в момент оплаты** (READY_STOCK): позиция → предложение перевести в
  warm/manual или возврат средств.
- **Нет свободного оператора/ресурса**: задача остаётся в queued/on_hold; ETA
  корректируется; при длительной задержке — уведомление и опция возврата.
- **Брак при QC**: rework (повтор этапов) или замена ресурса; покупатель не получает
  дефектный комплект.
- **Частичная выдача**: часть позиций выдана (stock), часть в работе (warm) →
  `partially_delivered`; заказ закрывается по мере готовности позиций.
- **Отмена покупателем до старта работ**: возможна для queued (по политике) с возвратом;
  после старта прогрева — по согласованию/через поддержку.
- **Гарантийная замена**: в окне гарантии → задача Rework/новый ресурс или возврат.
- **Просрочка ETA**: эскалация; покупателю — новая оценка; при срыве — приоритетная
  обработка или возврат.

## Диаграмма (сводно)

```
Покупатель                Платформа/Воркеры                 Операторы (админка)
    │  оформляет заказ          │                                   │
    ├──────────────────────────►│ проверка наличия/резерв           │
    │  оплачивает               │ списание (ledger), paid           │
    │◄──── ETA + статус ────────┤                                   │
    │                           │ READY_STOCK: reserved→delivered   │
    │                           │ MADE_TO_ORDER: создаёт Warming Job ├──► Queued
    │◄──── статусы/этапы ───────┤◄─── обновления статуса ───────────┤ Assigned→In progress→QC→Ready
    │  получает комплект (Vault)│ Delivered + уведомление           ├──► Delivered
    │  (опц.) запрос замены     │ Rework/Refund                     │
```
