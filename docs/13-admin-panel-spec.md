# 13 — Полная админ / бэк-офис: спецификация

Это не «пара экранов», а полноценная операционная система управления магазином,
инвентарём, прогревом и выдачей. Ниже — роли, модули, экраны и воркфлоу.

## Роли и права (RBAC)

| Роль | Доступ |
|------|--------|
| **Owner** | Всё, включая настройки, роли, интеграции, опасные действия. |
| **Admin** | Всё, кроме передачи владения; управление персоналом. |
| **Manager** | Каталог, цены, планы прогрева, распределение задач, финансы, отчёты, инвентарь. |
| **Operator** | Очередь прогрева/провижининга, инвентарь (выдача ресурсов), сборка комплекта. Без финансов. |
| **Support** | Тикеты, просмотр заказов/пользователей, инициирование замен/возвратов (по политике). |

Каждое чувствительное действие логируется в **AuditLog** (кто/что/когда/до→после).

## Карта модулей

```
Dashboard · Orders · Warming queue (Kanban) · Operator workspace · Products & Bundles
· Warming plans · Stock (ready accounts) · Proxy inventory · Octo profiles · Users
· Finance · Promo · Tickets · Reports/Analytics · Staff & roles · Audit log · Settings/Integrations
```

## Модули (детально)

### 1. Dashboard
- KPI: выручка, заказы, конверсия, средний чек; **операционные**: в очереди прогрева,
  в работе, на QC, просрочено по SLA, ready-к-выдаче, открытые тикеты.
- Виджеты: очередь по goal, загрузка операторов, ближайшие дедлайны, последние события.
- Фильтр по периоду; экспорт.

### 2. Orders (заказы)
- Таблица всех заказов: номер, покупатель, состав, сумма, тип выдачи (stock/warm),
  статус, ETA/факт, оператор. Фильтры (тип, статус, goal, период, просрочка).
- Карточка заказа: позиции, привязанный Warming Job (если есть), ресурсы (аккаунт/
  прокси/профиль), история статусов, оплаты/ledger, связанные тикеты.
- Действия: выдать вручную (stock), отменить, вернуть (refund→ledger), инициировать
  замену, переназначить оператора. Необратимое — через danger-confirm.

### 3. Warming queue — Kanban (ядро операционки)
- Колонки: **Queued → Assigned → In progress → QC → Ready → Delivered** (+ On hold,
  Failed/Rework).
- Карточка задачи: заказ, goal/план, ETA/SLA-таймер (зелёный/жёлтый/красный),
  оператор, текущий этап, прогресс (n/N этапов).
- Действия: назначить/переназначить, взять в работу, перевести этап, поставить на
  hold, отправить на QC, вернуть на доработку. Drag-n-drop между колонками.
- Фильтры: goal, оператор, просрочка, тариф. Быстрый поиск по номеру заказа.

### 4. Operator workspace (рабочее место оператора)
- «Мои задачи» + «свободные к взятию» (по goal/скиллу).
- Экран выполнения Job: пошаговый чек-лист плана прогрева; на каждом этапе — отметка
  выполнения, заметки, вложения/артефакты.
- Привязка ресурсов прямо из задачи: выбрать прокси из инвентаря, создать/привязать
  Octo-профиль, внести данные аккаунта (шифруются).
- Кнопка «Собрать комплект» → предпросмотр Bundle → «На QC».
- SLA-индикатор и оставшееся время до ETA.

### 5. Products & Bundles (товары и комплекты)
- CRUD категорий, товаров, вариантов (SKU). На варианте:
  - `fulfillmentType`: READY_STOCK / MADE_TO_ORDER;
  - `goal` (для warm), `warmingPlanId`, `tier`;
  - **конструктор комплекта**: какие компоненты включены (ACCOUNT/PROXY/OCTO_PROFILE/
    RECOVERY/SECRETS/GUIDE/WARRANTY) и их параметры (тип/гео прокси, срок, тип профиля);
  - цена, надбавки, гарантийное окно, переводы EN/RU, публикация/черновик.
- Предпросмотр карточки товара и авто-расчёт ETA из плана.

### 6. Warming plans (планы прогрева)
- CRUD планов: привязка к goal/tier, список этапов (название, ожид. длительность,
  чек-лист, требуемые компоненты), правила QC.
- Версионирование планов (изменение не ломает уже идущие задачи).
- ETA = сумма длительностей этапов.

### 7. Stock — готовые аккаунты (READY_STOCK)
- Список единиц по вариантам: available/reserved/sold; импорт CSV/TXT с отчётом по
  строкам; просмотр payload — под доступом и аудитом (шифрование).

### 8. Proxy inventory (инвентарь прокси)
- Таблица прокси: тип, гео, провайдер, срок, статус (available/assigned/expired),
  привязка. Добавление вручную/импортом. Массовые операции. Оповещения об истечении.
- Выдача: резерв/привязка к заказу при провижининге.

### 9. Octo profiles (реестр профилей)
- Реестр профилей: идентификатор в Octo, связанный прокси/аккаунт, статус
  (draft/ready/delivered), ссылка на экспорт/шеринг (шифрование ссылок/артефактов).
- Создаются вручную оператором; платформа хранит связки и ссылки (не сам браузер).

### 10. Users (покупатели)
- Список/поиск, карточка: заказы, баланс/ledger, тикеты, флаги риска. Действия:
  блокировка (confirm, отзыв сессий), корректировка баланса (обязательная причина → ledger+audit).

### 11. Finance
- Пополнения, списания, возвраты; сходимость по ledger; себестоимость (прокси/профили/
  время оператора — опц.), маржинальность заказов; экспорт.

### 12. Promo (промокоды) — по мере надобности
- Создание/аналитика: процент/фикс, лимиты, сроки, привязка к категориям/goal.

### 13. Tickets (поддержка)
- Очередь тикетов, привязка к заказам/задачам, статусы, назначение на support,
  внутренние заметки, макросы-ответы.

### 14. Reports / Analytics
- Продажи по категориям/goal, время выдачи (план vs факт), SLA-соблюдение, загрузка
  операторов, доля замен/возвратов, LTV/повторные покупки, топ-товары.

### 15. Staff & roles
- Управление сотрудниками, роли/права (RBAC), скиллы операторов (по goal),
  доступность/смены (для распределения задач).

### 16. Audit log
- Полный журнал действий персонала: фильтры по актору/сущности/типу действия, diff.

### 17. Settings / Integrations
- Общие настройки магазина, языки/переводы, шаблоны уведомлений (email/in-app),
  крипто-эквайринг, (в будущем) API Octo/провайдера прокси, ключи шифрования (KMS),
  политики гарантий/возвратов, роли и лимиты.

## Сквозные требования админки
- **Отзывчивость**: десктоп-first, но работает на планшете; таблицы со скроллом.
- **Массовые операции**: выбор строк, bulk-действия (импорт, назначение, экспорт).
- **Права на уровне действий**: кнопки/маршруты скрыты/заблокированы по роли.
- **Аудит и подтверждения**: опасные и денежные действия — danger-confirm + лог.
- **Реальное время (желательно)**: обновление очередей/статусов без перезагрузки
  (websocket/поллинг) — важно для операторской работы.
- **Поиск везде**: глобальный поиск по заказам/пользователям/ресурсам.
- **i18n админки**: интерфейс EN/RU.

## Операторский поток «под ключ» (сводно)
```
Оплата warm-заказа → Job в Queued → менеджер/авто назначает оператора → оператор берёт →
идёт по этапам плана (привязывает прокси, создаёт Octo-профиль, оформляет аккаунт) →
собирает Bundle → QC → Ready → Delivered (в Vault покупателя) → гарантийное окно.
Просрочка SLA → эскалация. Брак → Rework/Refund. Всё в аудите.
```
